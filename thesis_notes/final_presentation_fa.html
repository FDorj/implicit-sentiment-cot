Accuracy

معنی: درصد نمونه‌های test که label درست گرفته‌اند. دلیل استفاده: یک معیار ساده و مستقیم از درست‌بودن کلی سیستم می‌دهد.

Macro-F1

خود F1 میانگین هارمونیک Precision و Recall است: F1 = 2 * P * R / (P + R). در Macro-F1، F1 هر کلاس جدا حساب می‌شود و میانگین برابر گرفته می‌شود.

دلیل این انتخاب

در ISA معمولاً باید target، aspect، clue ضمنی و sentiment به هم وصل شوند. THOR-style reasoning این فرض‌های میانی را قابل دیدن می‌کند و بعداً برای diagnosis قابل استفاده می‌گذارد.

مثال trace

Target -> aspect -> implied opinion -> label. در مثال OS X Mountain Lion، trace باید متوجه شود که target فقط ذکر شده و opinion خاصی درباره آن وجود ندارد؛ پس label باید neutral بماند.

منطق پیاده‌سازی‌شده

controller، direct label، THOR label، diagnostic label، error type و confidence را می‌خواند و بعد یک ترتیب تصمیم‌گیری محافظه‌کارانه را اجرا می‌کند.

1. نرمال‌سازی

قبل از تصمیم، labelها، error type و confidence نرمال می‌شوند.

2. حفظ توافق

اگر Direct و THOR هم‌نظر باشند، همان label مشترک نگه داشته می‌شود.

3. قبول correction

اگر error type قابل اصلاح باشد و confidence به threshold برسد، diagnostic label پذیرفته می‌شود.

4. fallback

در غیر این صورت fallback استفاده می‌شود؛ معمولاً Direct، و بعد در صورت نیاز THOR یا diagnostic.

Pipeline نهایی: هر نمونه چطور به یک label می‌رسد؟

Input

Error-Type Reflection

disagreement و trace را می‌خواند و error_type، diagnostic_label و confidence می‌دهد.

Controller

از Direct، THOR و diagnostic signal یک label میانی محافظه‌کارانه و decision log می‌سازد.

Train-Calibrated Selector

روی train یاد می‌گیرد برای هر profile کدام source بهتر است: direct label، error type، confidence و domain.

Final Label

selected_prediction را خروجی می‌دهد و source نهایی را از Direct، THOR یا diagnostic انتخاب می‌کند.

نقطه تصمیم نهایی

controller اصلاح‌های diagnostic را توضیح و محدود می‌کند؛ اما final selected pipeline با train-calibrated selector منبع واقعی خروجی را انتخاب می‌کند. در اجرای validate‌شده overall، sourceها این بودند: Direct: 1,566، THOR: 621، Diagnostic: 1.

Scope

چه چیزی عمداً claim نشد؟

چرا Qwen3 8B اینجا fine-tune نشد؟

Ollama بیشتر inference-oriented است. fine-tuning این مدل به setup جدا مثل LoRA/QLoRA، GPU مناسب، validation دقیق و داده آموزشی بیشتر از 1,746 نمونه ISA train نیاز دارد.

Flan-T5 ارزیابی نشد

اسکلت runner مربوط به HuggingFace وجود دارد، اما نتایج Flan-T5 جزو آزمایش‌های validate‌شده فعلی نیست.

مقایسه API انجام نشد

مدل‌های OpenAI API یا APIهای خارجی دیگر در فایل‌های نتیجه فعلی نیامده‌اند.

THOR-style implementation

پروژه یک pipeline سازگار با ایده THOR-style دارد، نه reproduction دقیق official codebase.

وضعیت fine-tuning

وزن‌های Qwen3 8B تغییر نکردند. contribution فعلی طراحی pipeline است: prompting، diagnostic reflection، controller logic و train-calibrated source selection.
